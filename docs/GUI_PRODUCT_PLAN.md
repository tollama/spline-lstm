# GUI_PRODUCT_PLAN — Spline-TSFM 경량 크로스플랫폼 GUI 제품 계획

## Standard Handoff Format

### 1) 요청/목표
- 역할: Project Manager (GUI 제품 계획 수립)
- 프로젝트: `~/spline-lstm`
- 목표: **스마트폰/태블릿/개인 노트북에서 실행 가능한 경량 GUI**의 MVP 계획을 확정하고, Alpha/Beta 릴리즈 게이트를 정의한다.
- 핵심 제약:
  - 단일 코드베이스 기반(웹 우선)으로 디바이스별 UI 대응
  - 로컬 노트북 + 모바일 브라우저 환경에서 실사용 가능
  - 모델 추론/시나리오 실행 상태 및 로그 가시성 확보

---

### 2) 사용자 시나리오 (모바일/태블릿/노트북)

#### 2.1 모바일(스마트폰) 시나리오 — “빠른 상태 확인 + 즉시 실행”
- 사용자: 현장 운영자/개인 사용자
- 상황: 이동 중 최근 데이터셋으로 예측 시나리오를 재실행하고 결과 트렌드를 즉시 확인
- 사용자 흐름:
  1. 앱/웹 진입 → 최근 프로젝트 자동 선택
  2. 데이터 소스(최근 파일/샘플 데이터) 1탭 선택
  3. 시나리오 Preset(기본/보수/공격) 중 1개 선택 후 실행
  4. 결과 카드(핵심 지표 + 미니 차트) 확인
  5. 실패 시 로그 요약(에러 3줄 + 재시도 버튼) 확인
- UX 요구:
  - 세로 화면 기준 단일 컬럼 레이아웃
  - 버튼/입력 최소화(탭 중심)
  - 결과 요약 우선, 상세는 드릴다운

#### 2.2 태블릿 시나리오 — “비교 분석 + 협업 리뷰”
- 사용자: 분석 담당자/팀 리드
- 상황: 회의 중 2개 시나리오를 나란히 비교하고 변동 구간을 설명
- 사용자 흐름:
  1. 데이터 선택(파일/최근 실행) 후 파라미터 슬라이더 조정
  2. 시나리오 A/B 동시 실행
  3. 분할 화면에서 예측 라인/오차 지표/구간 하이라이트 비교
  4. 실행 로그 타임라인에서 경고 이벤트 확인
  5. 결과 스냅샷 저장
- UX 요구:
  - 가로 모드 최적화(2패널)
  - 차트 인터랙션(줌, 구간 선택)
  - 로그/결과 동시 조회

#### 2.3 개인 노트북 시나리오 — “심화 설정 + 반복 실험”
- 사용자: 개발자/파워 유저
- 상황: 실험용 파라미터를 세밀 조정해 반복 실행하고 로그 기반으로 튜닝
- 사용자 흐름:
  1. 로컬 데이터 파일 선택(CSV/NPZ)
  2. 시나리오 설정(윈도우, horizon, seed, model variant) 변경
  3. 실행 큐에 2~5개 배치 실행
  4. 결과 시각화(예측/실측/오차) + 실행별 로그 비교
  5. 결과 export(JSON/PNG)
- UX 요구:
  - 고급 옵션 패널(접기/펼치기)
  - 다중 실행 상태표(queued/running/done/failed)
  - 로그 필터링(레벨/키워드)

### 3) MVP 기능 범위 (필수/제외)

#### 3.1 MVP In-Scope (필수)
1. **데이터 선택**
   - 로컬 파일 업로드(CSV, NPZ)
   - 최근 실행 데이터 목록(최대 10개)
   - 스키마 기본 검증(필수 컬럼/shape)
2. **시나리오 실행**
   - Preset 실행(기본 3종)
   - 고급 옵션 최소 셋(window, horizon, seed)
   - 실행 상태 표시(ready/running/success/fail)
3. **예측 결과 시각화**
   - 시계열 라인 차트(실측 vs 예측)
   - 핵심 지표 카드(MAE/RMSE/MAPE 중 최소 2개)
   - 구간 확대(줌/팬) 및 최근 실행 비교 1건
4. **로그**
   - 실행 로그 스트림(시간순)
   - 로그 레벨 표시(INFO/WARN/ERROR)
   - 오류 시 요약 메시지 + 재실행 버튼

#### 3.2 MVP Out-of-Scope (제외)
- 사용자 계정/권한 관리
- 멀티 사용자 동시 편집
- 대규모 분산 실행/클러스터 스케줄링
- 알림 푸시/이메일 자동 발송
- 고급 리포트 빌더(커스텀 템플릿)

#### 3.3 기술 방향 (실행 중심)
- 프론트엔드: 반응형 웹(PWA 우선) + 모바일 브라우저 대응
- 백엔드: 기존 `spline-lstm` 실행 스크립트 래핑 API
- 통신: 비동기 실행 job id + polling/websocket 중 1개 확정
- 저장: 로컬 파일 기반 + 최근 실행 메타 인덱스(JSON)

### 4) 릴리즈 단계 (Alpha/Beta)

#### 4.1 Alpha (내부 검증, 2주)
- 목표: “핵심 플로우가 끊기지 않고 실행되는가” 검증
- 범위:
  - 데이터 선택 → 시나리오 실행 → 결과 시각화 → 로그 확인 end-to-end
  - 모바일/태블릿/노트북 기본 화면 깨짐 없음
- 품질 게이트:
  - 치명 오류(Crash) 0건
  - 실행 성공률 90%+
  - 주요 사용자 여정 3종(모바일/태블릿/노트북) 수동 테스트 PASS
- 산출물:
  - Alpha 테스트 리포트
  - 알려진 이슈 목록(우선순위 포함)

#### 4.2 Beta (파일럿 사용자, 3~4주)
- 목표: “실사용성 + 경량 성능” 확인
- 범위:
  - Alpha 기능 안정화
  - 결과 비교 UX 개선(태블릿/노트북)
  - 로그 검색/필터 고도화
- 품질 게이트:
  - P1 이상 결함 0건
  - 주간 재사용률(파일럿 사용자 기준) 60%+
  - 사용성 피드백 평균 4.0/5.0+
- 산출물:
  - Beta 운영 리포트
  - 정식 전환(Go/No-Go) 판정 문서

### 5) 성능/경량 목표 (초기 로딩, 메모리)

#### 5.1 디바이스 공통 목표
- 초기 로딩(첫 화면 인터랙션 가능 시점):
  - 모바일: **3.0초 이내**
  - 태블릿: **2.5초 이내**
  - 노트북: **2.0초 이내**
- 메모리(프론트엔드 런타임 기준):
  - 모바일 브라우저: **250MB 이하**
  - 태블릿 브라우저: **350MB 이하**
  - 노트북 브라우저: **500MB 이하**

#### 5.2 실행 중 성능 목표
- 시나리오 실행 시작 응답(버튼 클릭→상태 running): **1.0초 이내**
- 결과 차트 최초 렌더: **2.0초 이내** (기본 길이 데이터 기준)
- 로그 갱신 지연: **1.0초 이내**

#### 5.3 경량화 실행 항목
- 번들 분할(code splitting) 및 지연 로딩
- 차트 데이터 다운샘플링(모바일 우선)
- 불필요 라이브러리 제거(대체 가능한 경량 모듈 사용)
- 로그 가상 스크롤 적용(대량 로그 대응)

### 6) 완료 기준 (Go/No-Go)

#### 6.1 Go 기준 (모두 충족)
1. 기능 기준
   - MVP In-Scope 4개 영역 기능 동작 확인(PASS)
2. 품질 기준
   - 치명 버그 0건, P1 버그 0건, P2 버그 3건 이하
3. 성능 기준
   - 5장 목표의 초기 로딩/메모리/렌더 기준 충족률 90% 이상
4. 사용자 검증 기준
   - 모바일/태블릿/노트북 각 3명 이상 시나리오 테스트 완료
   - 태스크 성공률 85% 이상

#### 6.2 No-Go 기준 (하나라도 해당 시)
- 핵심 플로우 중 1개라도 재현성 있게 실패
- 모바일 메모리 상한 초과가 반복 발생
- 데이터 선택/실행/결과/로그 중 1개 영역 미완성
- 파일럿 피드백에서 Blocker 이슈 지속(수정 일정 불확정)

#### 6.3 조건부 Go (예외 승인)
- P2 이하 비치명 이슈만 잔존하고,
- 우회 절차(runbook)가 존재하며,
- 2주 내 수정 일정과 담당자 확정 시

### 7) 실행 계획 (Owner/Timeline)

| 주차 | 핵심 목표 | 세부 작업 | 담당 |
|---|---|---|---|
| W1 | MVP IA/와이어 확정 | 디바이스별 레이아웃/플로우 확정, API 계약 정의 | PM + FE + BE |
| W2 | 기능 구현 1차 | 데이터 선택/실행 상태/기본 차트/기본 로그 구현 | FE + BE |
| W3 | 기능 구현 2차 | 고급 옵션 최소셋, 오류 처리, 반응형 개선 | FE + BE |
| W4 | Alpha 게이트 | 3디바이스 시나리오 테스트, 결함 수정, Alpha 판정 | PM + QA |
| W5~W6 | Beta 파일럿 | 파일럿 운영, 성능 최적화, 피드백 반영 | PM + FE + BE + QA |
| W7 | 최종 판정 | Go/No-Go 회의, 런북/릴리즈 노트 정리 | PM |

### 8) 리스크 및 대응
- 리스크: 모바일 브라우저 메모리 부족
  - 대응: 차트 포인트 제한 + 단계적 로딩 + 로그 가상화
- 리스크: 실행 엔진 응답 지연
  - 대응: 비동기 job 큐 + 진행률 이벤트 + 타임아웃/재시도
- 리스크: 디바이스별 UI 불일치
  - 대응: 공통 컴포넌트 + 브레이크포인트별 스냅샷 테스트
- 리스크: 파일 포맷 변형 입력 증가
  - 대응: 업로드 전 스키마 프리체크 + 오류 템플릿 안내

### 9) 인수인계 메모
- 본 문서는 GUI MVP의 제품/실행 기준선(baseline)이다.
- 다음 액션:
  1. FE/BE가 `W1` 착수 전 API 계약서 초안 작성
  2. QA가 디바이스별 시나리오 테스트 케이스(모바일/태블릿/노트북) 작성
  3. PM이 Alpha/Beta 게이트 점검표를 별도 문서로 분리(`docs/GUI_RELEASE_GATE.md` 권장)
- 의사결정 필요사항:
  - 실행 상태 업데이트 방식(polling vs websocket)
  - 차트 라이브러리 최종 선정(경량 우선)
  - 모바일 최소 지원 브라우저 버전
