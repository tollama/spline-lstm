# GUI_PHASE_PLAN — GUI 개발 마스터 플랜 (Phase 1~5)

## Standard Handoff Format

### 1) Request / Scope
- 역할: Project Manager
- 프로젝트: `~/spline-lstm`
- 목표: GUI 개발을 **Phase 1~5 순차 게이트 방식**으로 운영하기 위한 공식 실행 계획 수립
- 필수 결과:
  1. Phase 1~5 정의(목표/산출물/DoD/기간)
  2. 각 Phase 진입/종료 Gate
  3. 역할별 책임(PM/Architect/Coder/Reviewer/Tester)
  4. 중간 진행 보고 포맷(사용자 공유용)
  5. 리스크 및 대응

---

## 2) 실행 원칙 (운영 기준선)
1. **Gate 기반 진행**: 이전 Phase Exit Gate PASS 없이는 다음 Phase 시작 금지
2. **증빙 우선**: “구현 완료”가 아니라 문서/테스트/실행 로그 증빙으로 완료 판정
3. **역할 분리**: Architect(설계) → Coder(구현) → Reviewer(품질) → Tester(검증) → PM(통합 판정)
4. **주간 고정 리듬**: 계획(월) → 구현(화~목) → 검증/판정(금)
5. **Blocker 즉시 에스컬레이션**: 24시간 이상 정체 이슈는 PM이 우선순위 재조정

## 3) Phase 1~5 정의 (목표/산출물/DoD/기간)

| Phase | 목표 | 핵심 산출물 | DoD (Definition of Done) | 권장 기간 |
|---|---|---|---|---|
| Phase 1. 요구사항/와이어프레임 확정 | 사용자 시나리오와 화면 구조를 확정해 구현 기준선 수립 | `docs/GUI_PRODUCT_PLAN.md` 업데이트, 화면 IA/플로우, 와이어프레임, API 초안 계약 | ① 핵심 유저플로우(데이터 선택→실행→결과→로그) 승인 ② 모바일/태블릿/노트북 화면 구조 확정 ③ API 인터페이스 초안 합의 | 1주 |
| Phase 2. UI Shell + 실행 파이프라인 MVP | 기본 GUI 화면과 백엔드 실행 연결 최소 기능 완성 | 기본 레이아웃, 실행 트리거/상태표시, 결과 차트 초안, 로그 패널 초안, MVP 데모 | ① MVP 필수 플로우 end-to-end 동작 ② 치명 오류(Crash) 0건 ③ 핵심 경로 수동 시연 성공 | 1~2주 |
| Phase 3. 데이터/결과/로그 고도화 | 데이터 검증·오류 처리·결과 비교·로그 가시성 강화 | 업로드 검증, 에러 UX, 결과 비교 UI, 로그 필터/검색, 테스트 케이스 확장 | ① 실패 시 복구 경로(재시도/가이드) 제공 ② 결과 비교 기능 동작 ③ 기능 테스트 PASS율 95%+ | 1~2주 |
| Phase 4. 품질/성능 안정화 | 반응형·성능·회귀 안정성을 릴리즈 수준으로 개선 | 성능 리포트, 회귀 테스트, 디바이스 호환성 리포트, 운영 Runbook | ① 성능 목표 충족률 90%+ ② P1 결함 0건 ③ 회귀 테스트 자동화 안정 통과 | 1주 |
| Phase 5. 파일럿/출시 게이트 | 사용자 파일럿 검증 후 Go/No-Go 판정 | 파일럿 결과 리포트, 결함 정리, 최종 릴리즈 노트, 운영 체크리스트 | ① Go 기준 충족(기능/품질/성능/사용성) ② Blocker 0건 또는 승인된 예외만 존재 ③ 최종 승인 서명 완료 | 1주 |

> 총 권장 일정: **5~7주** (팀 가용성/결함량에 따라 변동)

## 4) Phase별 Entry / Exit Gate

### Phase 1 Gate
- **Entry Gate**
  - 프로젝트 목표/범위(필수/제외) 문서화
  - 핵심 이해관계자(PO/PM/개발/QA) 참여자 확정
- **Exit Gate (PASS 기준)**
  - 와이어프레임/플로우 승인
  - MVP 기능 목록 확정(In/Out Scope)
  - API 계약 초안 리뷰 완료(Architect 승인)

### Phase 2 Gate
- **Entry Gate**
  - Phase 1 Exit PASS
  - 개발 브랜치 전략/코딩 규칙/PR 규칙 합의
- **Exit Gate (PASS 기준)**
  - 데이터 선택→실행→결과→로그 흐름 구현 완료
  - MVP 데모 성공(핵심 시나리오 최소 3개)
  - Reviewer가 Must-Fix 0건으로 판정

### Phase 3 Gate
- **Entry Gate**
  - Phase 2 Exit PASS
  - 오류/품질 개선 우선순위 백로그 확정
- **Exit Gate (PASS 기준)**
  - 입력 검증/오류 UX/결과 비교/로그 필터 기능 완료
  - 테스트 케이스 확장 반영 및 테스트 리포트 제출
  - Tester 검증 PASS + 주요 결함 재현율/해결율 보고 완료

### Phase 4 Gate
- **Entry Gate**
  - Phase 3 Exit PASS
  - 성능 목표 수치(로딩/메모리/렌더) 합의
- **Exit Gate (PASS 기준)**
  - 성능 기준 충족률 90% 이상
  - 회귀 테스트 자동화 PASS
  - 디바이스 호환성 체크리스트 PASS

### Phase 5 Gate
- **Entry Gate**
  - Phase 4 Exit PASS
  - 파일럿 대상/기간/성공지표 확정
- **Exit Gate (PASS 기준)**
  - 파일럿 결과에서 Go 기준 충족
  - 릴리즈 노트/Runbook/운영 연락체계 준비 완료
  - PM 최종 승인(Go) 또는 보완 일정 포함 조건부 Go 문서화

## 5) 역할별 책임 (R&R)

| 역할 | 핵심 책임 | 산출물 책임 | Gate 관점 책임 |
|---|---|---|---|
| PM | 일정/범위/우선순위 관리, 의사결정 조율, 리스크 에스컬레이션 | Phase 플랜, 주간 상태보고, Gate 판정 기록 | 각 Phase Entry/Exit 최종 판정 및 승인 기록 |
| Architect | 기술 구조/계약/API/비기능요건 정의 | 아키텍처 문서, API 계약, 기술 의사결정 로그 | Entry 기술 준비도 확인, Exit 설계 준수성 확인 |
| Coder | 기능 구현, 리팩토링, 코드 품질 유지 | 구현 코드, PR, 변경 로그 | Exit 전 기능 완성도/기술부채 보고 |
| Reviewer | 코드/문서 품질 검토, 정책/규칙 준수 확인 | 리뷰 리포트, Must-Fix 목록 | Must-Fix 해소 여부로 Gate 통과/보류 의견 제시 |
| Tester | 테스트 전략/케이스/자동화/수동 검증 | 테스트 케이스, 실행 결과 리포트, 결함 티켓 | 품질 기준 충족 여부(PASS/FAIL) 공식 판정 |

### 운영 규칙 (핵심)
- PM은 **진행률보다 Gate 충족 증빙**을 우선 확인
- Reviewer/Tester가 FAIL 시 Phase 종료 불가
- Architect 변경 승인 없는 API 파괴적 변경 금지
