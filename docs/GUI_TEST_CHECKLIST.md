# GUI_TEST_CHECKLIST

## Standard Handoff Format

### 1) 요청/목표
- 프로젝트: `~/spline-lstm`
- 목표: 모바일/태블릿/노트북 환경에서 GUI 품질 검증 기준(체크리스트 + PASS/FAIL 판정 기준) 정리
- 대상 화면(가정):
  1. 실행 화면(파라미터 입력/실행 버튼)
  2. 진행/로그 화면(상태, 에러 메시지)
  3. 결과 화면(지표/리포트/아티팩트 경로)
- 핵심 플로우 범위: 실행 성공 / 실행 실패 / 결과 확인 / 재실행

---

### 2) 검증 전제/환경
- 브라우저: Chrome, Safari(모바일), Edge(노트북)
- 뷰포트 기준:
  - 모바일: `360x800`, `390x844`
  - 태블릿: `768x1024`, `834x1194`
  - 노트북: `1366x768`, `1440x900`
- 네트워크 조건(성능/복원력 확인):
  - 정상: Wi-Fi
  - 저속: Fast 3G 또는 네트워크 Throttling
- 데이터/실행 시나리오:
  - 정상 실행: `scripts/run_e2e.sh` 또는 동등 GUI 실행
  - 짧은 실행: `scripts/smoke_test.sh` 또는 동등 GUI 실행
  - 오류 유도: 필수 입력 누락, 잘못된 run_id/경로, 백엔드 미기동

---

### 3) 반응형 레이아웃 체크포인트 (모바일/태블릿/노트북)

#### 3-1. 공통 레이아웃
- [ ] 헤더/내비게이션이 뷰포트 밖으로 잘리지 않음
- [ ] 핵심 CTA(실행, 중단, 결과 보기)가 첫 화면 또는 1회 스크롤 내 접근 가능
- [ ] 카드/패널/테이블 간 겹침(overlap) 없음
- [ ] 텍스트 줄바꿈 시 레이아웃 깨짐 없음(라벨, 로그, 경로 문자열)
- [ ] 가로 스크롤이 필요한 컴포넌트(로그/테이블)는 의도된 영역 안에서만 발생

#### 3-2. 모바일 (360~430px)
- [ ] 1열 레이아웃으로 자연스럽게 전환(2열 고정 금지)
- [ ] 입력 폼 터치 타겟 높이 44px 이상
- [ ] 키보드 오픈 시 입력 필드/실행 버튼이 가려지지 않음
- [ ] 긴 파일 경로/에러 메시지 축약 표시 + 전체 보기 제공
- [ ] 하단 고정 버튼이 시스템 UI(홈 인디케이터)와 충돌하지 않음

#### 3-3. 태블릿 (768~1024px)
- [ ] 1~2열 혼합 레이아웃 전환이 자연스러움
- [ ] 좌/우 패널(설정/로그) 동시 노출 시 최소 폭 유지(가독성 확보)
- [ ] 세로/가로 회전 시 상태 유지(입력값, 진행 상태)
- [ ] 모달/드로어가 화면 중앙 정렬 및 닫기 동작 정상

#### 3-4. 노트북 (1366px+)
- [ ] 주요 정보(입력, 진행, 결과 요약) Above-the-fold 배치
- [ ] 테이블/차트/로그의 열 너비 자동 조정이 과도하지 않음
- [ ] 브라우저 확대/축소(80%, 125%)에서도 UI 손상 없음

---

### 4) 핵심 플로우 테스트 (실행/결과/오류)

#### [FLOW-01] 정상 실행 (Happy Path)
- 목적: 기본 실행부터 결과 확인까지 사용자 여정 검증
- 절차:
  1. 기본 파라미터 입력(run_id, epoch 등)
  2. 실행 버튼 클릭
  3. 진행 상태(로딩/진행률/로그) 확인
  4. 완료 후 결과 화면 이동
- 기대결과:
  - 상태가 `대기 -> 실행중 -> 완료`로 일관되게 전이
  - metrics(예: mae/rmse/mape/r2) 및 리포트 링크 노출
  - 실행 완료 시 중복 클릭 방지(버튼 비활성 해제 타이밍 정상)

#### [FLOW-02] 빠른 스모크 실행
- 목적: 짧은 경로 실행과 즉시 판정 UX 검증
- 절차: smoke 실행 전용 옵션/버튼으로 실행
- 기대결과:
  - 완료까지 UI 블로킹 없이 상태 업데이트
  - PASS 시 요약 배지/토스트 노출, 관련 산출물 링크 표시

#### [FLOW-03] 입력 오류 처리
- 목적: 사용자 실수에 대한 예방/복구 가능성 확인
- 절차: 필수값 누락, 잘못된 형식(run_id 특수문자 등) 입력 후 실행
- 기대결과:
  - 실행 전 클라이언트 검증으로 즉시 차단
  - 필드 단위 에러 메시지(무엇을 어떻게 고쳐야 하는지) 표시
  - 첫 오류 필드로 포커스 이동

#### [FLOW-04] 실행 실패 처리 (백엔드/런타임)
- 목적: 서버/러너 실패 시 오류 가시성 검증
- 절차: 백엔드 중단 또는 오류 조건(run_id mismatch 등)으로 실행
- 기대결과:
  - 사용자 친화적 오류 요약 + 상세 로그 확장 보기 제공
  - 재시도/설정수정/로그복사 액션 제공
  - 실패 상태가 성공 상태로 오인되지 않음

#### [FLOW-05] 결과 재조회/재실행
- 목적: 반복 작업 시 상태 일관성 검증
- 절차: 완료 후 동일/다른 파라미터로 재실행
- 기대결과:
  - 이전 결과와 신규 실행 결과 구분 표시(시간/run_id)
  - stale 데이터(이전 metrics) 잔존 없음

### 5) 성능 체크 (초기 로딩, 인터랙션)

#### 5-1. 초기 로딩
- [ ] First meaningful content 3초 이내(노트북/Wi-Fi 기준)
- [ ] 모바일 저속망(Fast 3G)에서 5초 이내 핵심 UI 렌더
- [ ] 로딩 중 스켈레톤/프로그레스 표시로 공백 화면 방지

#### 5-2. 인터랙션
- [ ] 버튼 클릭/탭 전환 반응 100ms~200ms 체감 범위
- [ ] 실행 상태 폴링/로그 업데이트 중 스크롤 끊김 최소화
- [ ] 입력 폼 타이핑 지연 없음(특히 모바일)
- [ ] 장시간 실행(>=2분) 중 메모리 급증/브라우저 멈춤 없음

#### 5-3. 안정성
- [ ] 네트워크 단절 후 복구 시 상태 재동기화 가능
- [ ] 새로고침 후 최근 실행 상태/결과 복원 정책이 일관됨

---

### 6) 접근성/가독성 기본 점검
- [ ] 텍스트 대비(본문/버튼/경고) WCAG AA 수준(일반 텍스트 4.5:1 이상 권장)
- [ ] 키보드만으로 핵심 동작 가능(Tab 이동, Enter 실행, Esc 닫기)
- [ ] 포커스 인디케이터가 시각적으로 명확함
- [ ] 아이콘 단독 버튼에 대체 텍스트(aria-label) 제공
- [ ] 상태 변화(성공/실패/로딩)가 색상 외 텍스트/아이콘으로도 전달됨
- [ ] 로그/에러 영역은 스크린리더가 읽을 수 있는 구조(role/aria-live) 적용
- [ ] 기본 폰트 크기 14px 이상, 본문 line-height 1.4 이상 권장

---

### 7) PASS/FAIL 기준

#### 7-1. 최종 판정 규칙
- **PASS:**
  1. 반응형 레이아웃 필수 항목(3장) Critical 0건
  2. 핵심 플로우(FLOW-01~05) 전부 성공
  3. 성능 기준(5장) 중 초기 로딩/인터랙션 임계치 충족
  4. 접근성/가독성 기본 항목 중 필수 항목 100% 충족
- **FAIL:**
  - 아래 중 하나라도 해당 시 FAIL
    - 실행 불가/결과 확인 불가 수준의 결함(Critical)
    - 모바일/태블릿에서 CTA 접근 불가
    - 오류 상태를 성공으로 오인하게 만드는 상태 표시 오류
    - 키보드 탐색 불가 또는 텍스트 판독 곤란 수준의 가독성 문제

#### 7-2. 결함 등급 정의
- **Critical:** 핵심 기능 사용 불가, 데이터 오인/손실 위험
- **Major:** 대체 경로는 있으나 주요 사용성 저하
- **Minor:** 기능은 가능하나 시각/문구/정렬 이슈

#### 7-3. 릴리즈 게이트 제안
- Critical 0, Major 0이면 릴리즈 가능
- Major 1건 이상이면 수정 후 재검증 필수

### 9) 인수인계 메모
- 본 문서는 GUI가 CLI 실행(`run_e2e.sh`, `smoke_test.sh`)을 감싸는 운영 UI를 가정해 작성함.
- 실제 화면 정보구조(IA)가 확정되면 FLOW ID 기준으로 케이스를 세분화(예: 로그인/권한/히스토리 조회) 권장.
- 자동화 전환 시 우선순위: `FLOW-01`, `FLOW-03`, `FLOW-04` (회귀 리스크가 큼).
